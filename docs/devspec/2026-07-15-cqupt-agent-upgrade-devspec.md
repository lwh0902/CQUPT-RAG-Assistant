# CQUPT Agent 能力升级 DevSpec

## 1. 目标与范围

将现有“重邮极客 Agent”从以本地 RAG 为中心的聊天应用，升级为可验证、可配置、可延续的校园知识助手。本规格覆盖五项能力：

1. 用户显式开启的真实联网搜索。
2. 校内知识库与网页证据的统一来源、融合和置信度输出。
3. 按账号跨设备持久化的模型参数设置。
4. 用户按需生成、但不写入记忆的当前会话总结。
5. 结构化、可覆盖、可审计的长期记忆系统。

本次不实现 MCP 适配器、外部事务工具或多代理规划器；它们以统一证据和服务边界为后续基础。现有 Function Calling 工具继续可用。

## 2. 现状与约束

- 后端为 FastAPI，聊天主入口是 `routers/chat.py`，通过 `/ws/chat` 进行流式输出。
- 本地知识库检索为向量检索、BM25 和 RRF 融合；`rag.py` 当前输出以拼接文本为主。
- 用户、会话和消息保存在 MySQL；长期摘要当前写入 Chroma 的 `chat_long_term_memory` 集合。
- 前端为 React、TypeScript、Zustand 和 Tailwind 风格工具类；聊天页为 `frontend/src/pages/ChatPage.tsx`。
- 现有代码中部分模型调用参数是硬编码的。用户配置只能影响最终回答生成，不能影响意图识别、查询改写、记忆提取和会话总结等内部控制任务。

## 3. 产品行为

### 3.1 联网搜索开关

聊天输入框上方提供带图标和 tooltip 的“联网搜索”二元开关，默认关闭。

- 关闭时：仅执行现有本地知识库检索；不发起任何网络搜索请求。
- 开启时：本地知识库检索与真实网络搜索并行执行，再统一融合证据。
- 开关只影响当前待发送消息。消息发送后，界面复位为关闭，避免用户无意持续消耗网络搜索配额。
- 前端流式过程须独立展示“检索校内知识库”“搜索互联网”“整理证据”三个状态。
- 网络检索必须是后端调用搜索供应商 API 的结果。模型训练知识、模型自行生成的 URL、没有实际 URL 的内容均不得标为网络来源。

### 3.2 回答、来源与置信度

每次完成回答都返回回答正文、来源列表、置信度等级、证据说明和不确定项。

- 来源区按“校内知识库”和“网络来源”分组。
- 网络来源至少显示标题、站点名、真实 URL、发布日期或检索时间和引用片段；点击在新标签页打开。
- 知识库来源显示文档名、页码（若有）和引用片段；沿用现有 PDF 预览能力。
- 当校内材料与网页内容冲突时，回答必须明确说明冲突；排序优先级为：校内正式文件、学校官网、教育主管部门或政府官网、权威机构、普通网页。
- 无可用证据时，回答必须说明“暂无法从已检索资料确认”，而不是伪造来源或给出高置信度结论。

### 3.3 模型设置

右上角提供设置入口，打开轻量设置面板。

- `temperature`：范围 0.0 至 1.0，步长 0.1，默认 0.3。
- `top_p`：范围 0.1 至 1.0，步长 0.1，默认 0.8；UI 中文名称为“多样性”。
- 首次打开时从服务端加载；滑块结束拖动后保存到服务端，并在当前页面同步显示保存结果。
- 同一用户在另一台设备登录后读取相同配置。
- 下一次新发起的回答使用最新配置；正在流式生成的回答不得被中途改变。

### 3.4 当前对话总结

会话头部提供“总结当前对话”图标按钮，仅在当前会话至少有两条消息时可点击。

- 点击后，由模型根据当前会话全部消息生成结构化摘要：`topic`、`confirmed_points`、`open_questions`、`next_actions`。
- 前端将摘要展示为可收起面板，并提供复制和重新生成操作。
- 总结不写入 MySQL 记忆表、不写入 Chroma、不参与后续回答上下文、不改变现有自动摘要行为。
- 生成失败、空会话和无权限访问时，显示可理解的错误状态，原消息保持不变。

### 3.5 结构化记忆

将长期记忆拆为不同生命周期和用途，避免把所有窗口摘要视为同一类事实。

| 类型 | 示例 | 持久化位置 | 是否进入下轮上下文 |
|---|---|---|---|
| 短期上下文 | 最近四轮原始消息 | MySQL `messages` | 是 |
| 情景记忆 | 过去讨论过的具体事项 | Chroma | 按语义召回 |
| 用户偏好 | 希望回答简洁 | MySQL | 是 |
| 用户事实 | 用户是软件工程专业 | MySQL | 是，低风险使用 |
| 任务状态 | 正在准备奖学金材料 | MySQL | 是 |
| 手动会话总结 | 用户点击生成的摘要 | 仅前端状态 | 否 |

稳定记忆只从用户明确表达、并对未来有价值的信息中提取。模型猜测、一次性情绪和低置信信息不得写入。

当同一 `memory_key` 有更新值时，旧记录标记为 `superseded`，新记录成为 `active`；不得直接丢弃旧值。所有创建、更新、覆盖和删除必须记录审计事件。

## 4. 系统设计

### 4.1 证据对象

本地知识库和网页搜索在进入回答阶段前统一为 `EvidenceSource`：

```json
{
  "id": "web:4c8a...",
  "source_type": "web",
  "title": "研究生奖助政策通知",
  "url": "https://example.edu.cn/notice/123",
  "site_name": "重庆邮电大学",
  "document_name": null,
  "page": null,
  "snippet": "……",
  "published_at": "2026-06-01",
  "retrieved_at": "2026-07-15T12:00:00+08:00",
  "relevance_score": 0.86,
  "authority_score": 0.95
}
```

`source_type` 仅允许 `knowledge_base` 或 `web`。网页来源必须有合法的 HTTP(S) URL；知识库来源必须具有文档标识或文档名称。

### 4.2 检索和生成流程

```text
用户发送消息
  -> 读取账号模型设置
  -> 判断闲聊或知识型提问
  -> 本地 Hybrid RAG 检索
  -> 若联网开关开启，同时调用 WebSearchProvider
  -> 标准化、去重、按权威性和相关性排序
  -> 计算置信度并检测冲突
  -> 将有限、带编号的证据传给模型
  -> 流式输出回答
  -> 返回 sources / confidence / uncertain_points
  -> 持久化本轮消息和既有自动长期摘要流程
```

搜索供应商封装为 `WebSearchProvider` 协议。第一期仅接入一个供应商，供应商名称、API 密钥、超时和最大结果数通过环境变量配置；业务代码不得直接依赖供应商响应格式。

网络搜索异常、限流、超时或零结果时，应保留本地检索结果并返回可诊断的不确定项。网络搜索不得阻塞超过配置的超时时间。

### 4.3 置信度

置信度由后端规则计算，模型不能自行声明等级。计算输入包括：证据相关性、权威性、独立来源数量、元数据完整度、来源冲突和仅摘要命中等信号。

| 等级 | 区间 | 含义 |
|---|---:|---|
| `high` | >= 0.80 | 高相关且有校内正式材料或多个可信来源一致 |
| `medium` | >= 0.55 | 有足够相关证据，但时间、完整性或来源数存在限制 |
| `low` | >= 0.30 | 证据较弱、仅摘要或出现未解决冲突 |
| `unknown` | < 0.30 | 没有可核验证据支撑 |

页面仅展示中文等级，不展示虚假的精确小数；API 仍保留 `confidence` 供测试和分析使用。

### 4.4 数据模型和接口

新增表：

```text
user_model_settings
  user_id (PK/FK users.id)
  temperature DECIMAL(2,1)
  top_p DECIMAL(2,1)
  updated_at DATETIME

agent_memories
  id (PK)
  user_id (FK users.id)
  session_id (nullable FK sessions.id)
  memory_type
  memory_key
  memory_value TEXT
  confidence DECIMAL(3,2)
  status (active | superseded | deleted)
  source_message_id (nullable FK messages.id)
  expires_at (nullable)
  created_at
  updated_at

memory_events
  id (PK)
  memory_id (FK agent_memories.id)
  event_type (created | updated | superseded | deleted)
  old_value TEXT (nullable)
  new_value TEXT (nullable)
  reason
  source_message_id (nullable)
  created_at
```

新增或变更的 API/消息契约：

```text
GET  /settings/model
PUT  /settings/model
POST /sessions/{session_id}/summary
WS   /ws/chat   # 新增 web_search_enabled 字段和来源/置信度终止事件
```

WebSocket 请求增加 `web_search_enabled: boolean`，默认 `false`。终止事件增加 `sources`、`confidence`、`confidence_level`、`evidence_summary` 和 `uncertain_points`。

所有新接口均使用现有认证方式，并验证会话归属当前用户。服务端对 `temperature`、`top_p`、会话 ID 和 URL 进行严格校验。

## 5. 前端与响应式规范

页面不新增营销卡片或固定宽度浮层。现有聊天工作台保持主界面优先。

| 视口 | 关键要求 |
|---|---|
| 手机：320-639 px | 侧栏默认收起；会话标题、联网开关、总结和设置可换行但不可重叠；来源卡单列；设置面板全宽或底部抽屉；输入区安全区内固定 |
| 平板：640-1023 px | 可显示紧凑侧栏；来源两列仅在每项可读时启用；设置面板不超出可视区 |
| 桌面：>=1024 px | 保留侧栏和居中消息列；头部操作在一行；来源区可使用两组并排布局 |

所有交互控件至少 44 x 44 CSS px 触控区域；文本不使用随视口缩放的字号；长标题、URL 和来源名必须截断或换行，不能挤压发送按钮或覆盖消息内容。按钮使用图标并提供 `aria-label` 和 tooltip。键盘焦点、暗色模式和加载/失败/禁用状态均为必需状态。

## 6. 测试与验收

每项功能必须先有自动化测试，再实现业务代码。新功能不得以“手工点过”为唯一验收依据。

### 6.1 后端测试

| 功能 | 必测用例 |
|---|---|
| 联网开关 | 关闭时 WebSearchProvider 零调用；开启时本地检索和网络检索都被调用；网络超时后仍返回本地回答和不确定项 |
| 证据 | 网页缺 URL 时被过滤；知识库缺文档标识时被过滤；重复来源合并；冲突来源被标记 |
| 置信度 | 高权威一致证据为 high；仅网页摘要不高于 low；无证据为 unknown；冲突降低分数 |
| 设置 | 未保存配置返回默认值；合法值可保存并读取；越界值返回 422；用户无法读取或修改他人配置 |
| 会话总结 | 有权限的有效会话返回结构化摘要；空会话拒绝；跨用户会话返回 403；调用后 `agent_memories` 和 Chroma 无新增记录 |
| 记忆 | 明确事实创建 active 记录；同 key 新值覆盖旧值；旧记录变 superseded；审计事件完整；低置信推断不写入 |
| 流式协议 | 每个新增思考状态顺序正确；`end` 事件含完整回答元数据；异常事件不会泄露 API 密钥或内部堆栈 |

测试使用 pytest、FastAPI TestClient/异步客户端和可替换的假 LLM、假搜索供应商。不得在测试中调用真实模型或真实搜索 API。

### 6.2 前端测试

| 功能 | 必测用例 |
|---|---|
| 联网开关 | 默认关闭；点击改变本次发送 payload；发送后自动复位；流式时禁用重复操作 |
| 来源和置信度 | 正确分组；网页链接真实渲染；unknown 显示“不确定”；空来源不渲染空容器 |
| 设置 | 初始加载服务端值；滑块改变数值和请求 payload；保存失败回滚或明确提示；正在回答时提示仅影响下一轮 |
| 会话总结 | 无消息时禁用；成功展示结构化区块；复制可用；重新生成替换视图；失败提示不清空原对话 |
| 可访问性 | 图标按钮存在可访问名称；键盘可操作开关、滑块和弹层；焦点可见 |

测试使用项目现有前端测试工具；若尚未配置，首个实施任务必须补齐 Vitest、React Testing Library 和浏览器端到端测试命令。

### 6.3 响应式视觉回归

每次涉及聊天页、设置面板、来源展示或总结面板的前端改动，必须运行浏览器端到端截图检查，至少覆盖：

```text
320 x 568   小屏手机
390 x 844   主流手机
768 x 1024  平板
1440 x 900  桌面
```

每个视口验证：无水平滚动条、无文本/控件重叠、输入区可见、弹层完全可达、长来源不溢出、暗色模式可读。截图是自动化验收产物，不替代交互断言。

## 7. 实施顺序与完成定义

1. 建立证据对象、来源输出、置信度规则和对应后端/前端测试。
2. 接入联网开关和一个真实搜索供应商，完成并行检索、失败降级和端到端测试。
3. 实现账号级模型设置及跨设备持久化测试。
4. 实现不入记忆的会话总结及隔离测试。
5. 迁移到结构化记忆，保留旧 Chroma 摘要作为过渡兼容层，并完成覆盖和审计测试。

只有当所有新增单元/接口/端到端测试通过，且四个目标视口的截图检查无溢出、重叠和不可操作控件时，本规格对应的功能才可标记为完成。
